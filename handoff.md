# TaskProgress handoff

- Status: Viewer theme system and the .NET 9 Windows Launcher are complete and verified.
- Active claim: none.
- Viewer source: `viewer/index.html`, `viewer/assets/`; public deployments place Viewer one level below the site root and keep `reports/` at the site root.
- Viewer distribution: consumer workflows use sparse checkout for `viewer/`; Pages artifacts copy only `viewer/index.html` and `viewer/assets/`. CLI source, tests, builds, schemas, and internal documentation are neither checked out nor deployed. `notify-docs.yml` dispatches Viewer updates to the docs repository when its cross-repository token is configured.
- Viewer UX: the Viewer uses a compact, centered progress-dashboard layout. The title, overall percentage, progress bar, colored status summaries, filters, task cards, and work rows are vertically dense; every task card shows a top-right fraction without a task percentage. Developer cards expose `Next Step` immediately below the task summary and expand only for approach/direction details. The no-query localhost homepage loads a sanitized runtime catalog and lists every scope registered by `start`; static/public hosting falls back to the original link-first instructions.
- Progress precedence: when a task lists `completed_items` or `pending_items`, its card fraction and the overall project calculation derive from those visible items. Explicit `progress` is only the fallback for tasks without item lists, preventing stale metadata from contradicting the rendered rows.
- Viewer themes: the default `system` mode follows `prefers-color-scheme`; explicit light, dark, and custom modes persist in `localStorage`. Custom palettes expose eight semantic core colors, inherit status colors from a light/dark base, validate `#RRGGBB`, and warn below 4.5:1 text contrast. A pre-CSS bootstrap applies saved preferences before paint; theme settings remain browser-local and do not change the report schema.
- Scope rule: `?scope=<id>` resolves to `../reports/<id>/report.json`; explicit `?report=` wins. Scope never auto-loads a developer report. The same relative rule works for the source `viewer/`, nested public deployments, and the localhost root.
- Contracts: `schemas/report.schema.json`, `schemas/report.dev.schema.json`; the tracked public sample is under `reports/example/`. `examples/` is reserved for untracked local report fixtures.
- Primary Launcher source: `src/TaskProgress.Cli/`; it retains report validation, local scope store, browser opening, and `task-progress://` install/uninstall.
- Local hosting: C# no longer embeds Viewer assets or runs Kestrel. It discovers the repository's `viewer/` root (or an explicitly configured Viewer root), searches upward for the sibling LocalWebService, starts `localHost.py` in control mode when absent, or reuses the matching process on the selected port. `start` uses Win32 `CREATE_NEW_CONSOLE` for a dedicated visible Python console; existing services are reused without attempting to change their window mode.
- Control integration: public versioned health identifies the service; bearer-authenticated status verifies instance and Viewer root; POST files registers exact base/developer reports; authorized shutdown is exposed through `service stop`.
- Local URL: `http://127.0.0.1:8001/?scope=<id>` with explicit `&dev=../reports/<id>/report.dev.json` when the developer report exists. Multiple scopes share one process and port.
- Safety: state/health/instance/port/root must agree; stale state is removed only when its PID is no longer running; foreign ports and route conflicts fail without takeover; missing developer reports remove a previously registered developer route for that scope.
- Commands: `start`, direct report folder, `open --scope`, `scope add/remove/list`, `service status/stop`, and `protocol install/uninstall`. `start` validates every stored scope before service mutation, launches a dedicated console when the service is absent, registers all base/developer reports, removes stale report routes for scopes no longer registered, writes/registers a path-free catalog, and opens the Viewer root. `scope add <report-folder>` derives a kebab-case scope from the folder name (for example `BonghuoVR` -> `bonghuo-vr`), validates it against `report.json`, and retains the explicit-scope form. Help accepts `help`, `-h`, `-help`, and `--help`; its output is a compact command/option table without repeating the executable on every line. Default port is 8001; `--no-browser` and explicit `--port` are supported.
- Discovery overrides: `TASK_PROGRESS_VIEWER_ROOT`, `TASK_PROGRESS_LOCAL_WEB_SERVICE`, `TASK_PROGRESS_PYTHON`, `TASK_PROGRESS_HOME`, and `TASK_PROGRESS_SERVICE_STATE`.
- Published output: `Build/win-x64/task-progress.exe`, a self-contained single file. Documentation and ignore rules now use `Build` as the output root.
- Automated verification: Release build 0 warnings/errors; `dotnet format --verify-no-changes`; 22 Node Viewer tests, including theme normalization, storage, application, OS preference, and contrast coverage; `tests/TaskProgress.Cli.Tests` verifies folder-name scope derivation/persistence, sanitized catalog serving, and real-process integration reuses one PID for two scopes, serves both reports, performs authorized shutdown, and removes state.
- Manual HTTP verification: two scopes produced four registrations on one PID; both report URLs returned the correct `scope_id`; `service status` and `service stop` succeeded with no remaining listener.
- Browser verification: visible-console `start` smoke on temporary port 8123 loaded one stored scope and cleaned up state on authorized shutdown. The root catalog listed `bonghuo-vr` and its Developer link without a local path; following the base link loaded 10 tasks and 45% progress with no console warnings/errors. Theme QA verified OS-dark resolution in system mode, explicit dark switching, persistence across reload, custom color application/persistence, contrast warnings, cancel semantics, zero console warnings/errors, and a 390px custom dialog with no horizontal overflow. Earlier static Viewer QA verified the compact dashboard at 1000x900 and 390x844, equal project/overall-progress heading sizes, no horizontal mobile overflow, task fractions without percentages, an immediately visible Next Step, and a working approach/direction expansion.
- Registry was not modified during verification; `protocol install` remains an explicit user action.
