# 時間參考原型（Draft 0.2）

這個目錄保存 TaskProgress「時間參考」Draft 0.2 的資料契約、可執行範例、計算邊界與原型介面。正式 Viewer 與 CLI 已透過可選 `time.analysis.json` sidecar 接入；Launcher 分析器可從 `report.json` 及選用的 `time.config`、`time.estimates`、`time.events` 確定性產生快照。

## 原型目標

時間參考不是承諾完成日期，而是把目前已知條件轉成可追溯、可逐步校正的估算：

1. **可用時間**：標準配置為一天 24 小時，睡眠 8 小時、生活 8 小時、其他固定不可工作時間 0 小時，每位執行者可投入 8 小時。
2. **工作估算**：規模、技術難度、未知技術探索與計畫內容形成預估工時。
3. **執行校準**：實際事件累積後，才可用來校準預估；沒有樣本時因子為 `1.0`、信心為低，而且不自動調整。
4. **期限急迫度**：比較工作進度與從開始到交付前已消耗的可用時間，輸出綠／黃／紅所需的狀態。

## 四份旁掛資料契約

| 檔案 | 用途 |
| --- | --- |
| `time.config.json` | 標準時間配置、單一執行者數、期限、顯示與急迫度門檻 |
| `time.estimates.json` | task/item 的低值、最可能值、高值、輸入參數、contributors、方法、公式、人工確認、信心與估算理由 |
| `time.events.json` | 工作時段、狀態變更、阻塞與估算修訂的事實紀錄 |
| `time.analysis.json` | 分析器依上述資料產生的衍生快照，供介面讀取 |

四份資料以相同的 `scope_id` 關聯。item 估算必須引用穩定的 `task_id`／`item_id`；正式 report 同時接受舊版字串項目與可供時間分析對齊的 `{id, title}` 項目物件。

`examples/` 同時包含可對齊的 `report.json`，因此可直接重現公開快照：

```powershell
.\Build\win-x64\task-progress.exe analyze `
  .\experiments\time-reference\examples `
  --as-of "2026-07-22T17:00:00+08:00"
```

正式專案的 `open`／`start` 在發現任一時間輸入時會先自動更新分析；沒有輸入時不建立 sidecar。明確執行 `analyze` 則允許只有 report 的專案使用每個穩定 item 480 分鐘、低信心的缺省模式。

## 第一版計算契約

```text
名目每日容量 = 每位執行者每日容量 × executor_count
顯示天數 = ceil(校準後總估算工時 ÷ 名目每日容量)

有效樣本權重 w = n_eff ÷ (n_eff + k)
執行校準因子 = (1 - w) × 1.0 + w × 歷史樣本因子

時間進度 t = 已消耗可用容量 ÷ 開始至交付前的全部可用容量
進度壓力 = (1 - 工作進度 p) ÷ (1 - 時間進度 t)
```

- 第一階段只正式支援 `executor_count = 1`。欄位先保留，避免未來破壞資料格式。
- `execution_calibration.factor` 大於 `1.0` 表示實際完成速度高於基準，小於 `1.0` 表示較慢。
- `n_eff` 是依任務相似度與資料品質加權的有效樣本數；`k` 是 config 中的先驗等效樣本數。樣本少時靠近 1.0，樣本增加後才逐漸接近歷史結果。
- 尚無完成樣本時 `n_eff = 0`、權重為 0、校準因子為 `1.0`，不得只因沒有資料就任意扣除執行率。
- 沒有工程計畫的 item 預設使用 `480` 分鐘（8 小時）、低信心、且不虛構低／高範圍。
- 每日工作容量 `capacity_minutes_per_executor_day` 與無計畫 item 的預設工時 `unplanned_item_likely_minutes` 是獨立參數；範例同為 480 分鐘只是第一版預設。每日容量負責工作日換算與容量時間線，預設項目工時只負責補出一筆明確的低信心估算。
- item 可以提供 `low_minutes`／`likely_minutes`／`high_minutes`；主畫面只顯示 likely，範圍放在詳細面板。
- Draft 0.2 不再使用互斥 `source`。`contributors` 可同時包含人工估算、人工參數、AI 分析、歷史證據、系統預設與固定公式；`inputs` 另外保存參數值、單位與提供者。
- `analysis_method` 保存 AI／人工／系統選擇的方法與版本；`calculation` 保存可重算公式。AI 負責工程判斷，確定的算術由固定公式執行。
- `human_confirmed` 只代表人工是否接受最後結果。人工提供參數、人工直接估值和人工確認是三種不同事件，彼此不互相暗示。
- contributors 可以重疊，因此不能直接加總其工時。專案 `estimate_composition` 使用互斥的 `manual`／`mixed`／`ai`／`default` 結果模式，確保每份有效估算只計入一次。
- 每份 estimate 保留 `active` 與 `supersedes_estimate_id`，讓人工修正或重新分析新增版本而非覆寫歷史。
- 原型以 ISO weekday `1..7`（週一到週日）表示可產生容量的工作日；沒有工作日設定時，分析器才退回日曆時間。
- `delivery_at` 是不可延後的精確交付界線。例如 `2026-08-01T00:00:00+08:00` 表示最後可工作時間在 7/31，8/1 一開始即停止。
- 第一版沒有可人工維護的獨立「剩餘工時」欄位。工作 session 只記錄實際投入與校準證據，不從總估算自動扣除；進度報告的「未完成工作量」只以校正後總工作量乘上最後回報的未完成比例即時衍生。

低／最可能／高的順序、每日配置總和、事件起訖與門檻順序等跨欄位關係，由未來分析器做語意驗證；JSON Schema 只負責結構與基本型別。

## 介面映射

主畫面維持極簡：

```text
8/1 交付 ● ›
75%
```

- `8/1 交付`：來自 deadline。
- 圓形燈號的綠／黃／紅來自 `deadline.urgency`：先反映未完成工程需求相對於剩餘真實容量的可行性，再以工作進度相對於時間進度的壓力補充趨勢；估算信心不混入燈號，外部不顯示判定文字。
- 點擊 Label 後先顯示即時摘要；展開詳細資訊後，以「評估流程／工程估算／工作容量」三個切換 Tab 分開總覽、估算與容量日曆。
- task 標題後顯示其子項目總預估工時，統一使用 `hr`。
- item 顯示小時；超過一天仍使用小時，並可點開查看 estimate 的理由、參與角色、輸入、方法與公式。

## 急迫度門檻

`deterministic-capacity-feasibility` v0.3 先比較剩餘工程需求與剩餘容量：

- `remaining / capacity > 1`：`critical`（紅）
- `0.8 < remaining / capacity <= 1`：至少 `at_risk`（黃）
- 容量使用率不高於 0.8 時，再使用進度壓力：`pressure <= 1.10` 為綠、`1.10 < pressure <= 1.50` 為黃、`pressure > 1.50` 為紅
- 已到交付時間但未完成：`critical`（紅）
- 已完成：`complete`（完成狀態色）
- 沒有交付期限：分析快照省略 deadline，Label 使用 Theme 預設顏色

低信心不會把綠色強制改成黃色；信心只在詳細面板揭露。

## 範例資料

`examples/` 示範一名執行者、每工作日 8 小時、於 2026-08-01 00:00 交付的分析：總預估 `2640` 分鐘（44 小時），剩餘估算 `2112` 分鐘。7/29 設為整日休假，因此從開始至交付前共有 `4320` 分鐘容量；分析時已消耗 `1440` 分鐘，剩餘 `2880` 分鐘，容量比率 0.7333，尚未觸發容量警戒。工作完成 20%，進度壓力為 `(1 - 0.2) / (1 - 0.3333) = 1.2`，因此整體急迫度仍為黃色。沒有完成樣本，因此校準仍為 1.0／低信心。

## 第一版實驗介面與正式整合

`demo/` 是保留用於快速試驗的靜態整合預覽。它直接引用
`viewer/assets/styles.css` 與正式 Viewer 的主要 DOM class，再用實驗 CSS 補上時間元件；
資料仍直接讀取
`examples/time.analysis.json`，呈現：

- 頁首「情境」選單預設顯示 `交付日未定`，可切回 `8/1 交付` 比較相同工程估算在有／無期限下的介面。切換只重建 Demo 記憶體資料，不修改 sidecar。
- `工時估算與截止日解耦` 是第四個穩定子項目，Demo 估算為 `12 hr`；加入後工程總預估為 `56 hr`，用來證明 item/task/project 工時不依賴 deadline。
- 無截止日模式保留 task/item 工時按鈕、估算來源、信心、公式、工程總工時與依工作進度衍生的未完成工時；專案膠囊改為中性的 `交付日未定 ›`，不顯示燈號、倒數、時間進度、容量或風險。期限區塊只填一半時採相同降級並顯示期限診斷。
- 可點擊的「8/1 交付 ● ›」工期摘要按鈕；它沿用主題選擇器／狀態標籤的膠囊弧度，但使用透明背景與中性外框，外部只保留交付日、圓形風險燈號與揭露箭頭。工程總預估工時及文字判定放在進度報告內。
- 工作進度與時間已使用比例。
- 點擊後開啟「進度報告」；預設只顯示距離交付、預估未完成工時（hr）、風險評估與最後回報。
- 專案詳細資訊分為 `評估流程`、`工程估算` 與 `工作容量` 三個共用內容區、可鍵盤切換的 Tab。流程 Tab 用雙軌節點呈現直接加總的未完成工時、剩餘容量、容量餘裕／缺口及現行風險，並明確說明 v0.3 容量缺口直接改為紅燈、容量使用率超過 80% 至少為黃燈；工程 Tab 包含總量、未完成量、時間／工作進度、壓力公式、本次計算、校準與估算組成；容量 Tab 包含每日分配、工作日、時區、交付前總／剩餘容量及休假例外。
- 8／8／8 基準在本機分析器計算一次：`24 hr - 睡眠 8 hr - 生活 8 hr - 其他固定不可工作 0 hr = 工作容量 8 hr`，再輸出每日容量時間線。Viewer 不會再次扣除。
- task 第一行顯示狀態燈號／文字與完成數，第二行在標題後顯示「約需 N hr」；三個 item 工時仍可點擊查看來源說明。
- item 列將無框工時放在名稱旁邊，將有邊框的 `進行中`／`待做` 狀態標籤固定在右側。
- item 面板預設顯示信心、共用的估算依據框，以及 `預估工時` 後方的 contributor 膠囊，例如 `[人工估算]`、`[人工參數] [AI 分析]` 或 `[預設]`；`N hr` 永遠靠該列最右。人工確認不混入 contributor 標籤。
- 詳細資訊顯示穩定 `item_id`、可換版 `estimate_id`、估算範圍、獨立人工確認、完整估算參與、輸入參數、分析方法、固定公式與 reference。
- 「詳細資訊」由 Demo 頁面共用同一個 `timeDetailsExpanded` flag；專案進度報告或任一 item 展開／收合後，關閉 dialog 再打開其他時間面板會沿用相同模式。重新載入頁面時才恢復預設收合。
- 具有人工輸入的 item 顯示「編輯」按鈕，可修改人工參數與獨立人工理由。按下「重新計算」後只執行 `calculation.algorithm_id` 指向、由 `demo/estimate-engine.js` 登錄的算法，不重新呼叫 AI、不執行任意公式字串，並建立下一個估算版本、重算 task/project 總和及把人工確認改為「否」。
- 「編輯」只在 `file://` 或 loopback（`localhost`、`127.0.0.1`、IPv6 loopback）環境出現；其他來源維持唯讀，且不讀取 Demo 的 localStorage 覆寫。此判定集中在 `demo/editing-policy.js`，未來可以替換成正式 edit capability，而不把權限綁在 `report.dev.json` 是否存在。
- 相同的本機政策也套用在 `工作容量` Tab。可編輯睡眠、生活、其他固定不可工作時間、工作日及 `日期 | 當日可工作 hr | 公開標籤` 形式的例外；工作容量由 24 小時扣除其他分配後衍生。公開 sidecar 只保存可公開標籤，不投影 config 中可能包含私人細節的 reason。儲存會重建期限前時間線並立即重算燈號，不呼叫 AI。
- Demo 覆寫保存在目前瀏覽器的 localStorage；重新整理會重新載入原始範例、套用人工覆寫並以算法重算。這只模擬動態資料流程，不代表正式 Viewer 取得寫入能力。
- `time.analysis.json` 採可選 sidecar 行為：HTTP 載入得到 404 時視為正常缺省，保留基礎進度畫面並隱藏所有工期、task 總工時與 item 工時；資料存在但無效時同樣降級並留下 console 診斷。Viewer 不在缺省時自行產生預設工時。
- sidecar 的 `deadline.schedule` 提供時區、本地工作時段、風險門檻與截止前逐日容量。Viewer 初始化／reload 時使用瀏覽器目前時間執行 `deadline-engine.js`；頁面開啟後每分鐘、分頁回到前景及 `pageshow` 時再次更新時間進度、壓力、邊界狀態與燈號。工作進度仍是最後發布的靜態快照，工程工時與 AI 分析不在公開 Viewer 重跑。
- v0.3 燈號會使用未完成項目估算與扣除 8／8／8、工作日及例外後的剩餘容量；預設項目工時也會直接進入容量可行性。容量足夠時才由工作進度與容量時間進度決定趨勢風險。
- 專案時間面板同時顯示 sidecar／報告的最後更新時間與 `deadline.evaluated_at` 本次風險計算時間；固定的範例說明已改為依即時計算結果及門檻生成。
- Demo 另提供進行中、已完成、受阻與已封存各一張狀態測試卡；正式 Viewer 樣式的篩選按鈕可切換只顯示相符卡片。額外三張測試卡不納入目前的六工作日分析快照。
- 響應式桌面／行動版配置與鍵盤可操作的原生 dialog。

`task_id`／`item_id` 是工作本身的穩定身分，不是工作類型；`estimate_id` 則辨識一次可被後續版本取代的估算。歷史資料可以作為 evidence contributor；等分析器真的計算相似度與權重後，才使用具版本的「歷史加權估算」方法。

可以直接雙擊 `demo/index.html`：`file:///` 模式使用內建的同值範例，避免瀏覽器阻擋本機 JSON。加入 `?time=none` 可預覽舊版／沒有 sidecar 時的降級畫面。從 repository root 啟動任意 loopback 靜態 HTTP server 後再開啟 `/experiments/time-reference/demo/`，則會讀取 `examples/time.analysis.json`；404 時不顯示時間功能，其他載入或格式錯誤也不退回預設資料。兩種本機模式都允許把 Demo 覆寫寫入瀏覽器 localStorage，但不會修改 report、Viewer 或任何時間資料檔。因為基礎樣式直接引用正式 Viewer，移動 `demo/` 或單獨複製 HTML 時必須一併保留 repository 目錄關係。

## 目前不做的事

- 不在 Viewer 內執行 AI、產生工程估算或回寫正式估算版本；Launcher 分析器負責確定性投影，Viewer 只驗證 sidecar 並在期限存在時重算隨時間變化的風險。
- 不把 Draft 時間欄位合併進基本 `report.json`；時間資料維持獨立 sidecar。
- 不建立個人 profile、排班、人員技能或團隊管理。
- 不讓 AI、歷史模板或預設值覆蓋人工資料；也不把人工提供參數誤判為已確認最後結果。
- 不處理多人並行效率、工作相依、資源競爭或最佳化排程。
- 不加入 NodaTime 或 OR-Tools 依賴。

未來開始實作日期／時區運算時，優先評估 NodaTime；只有在確定需要多人、相依與資源限制排程後才評估 OR-Tools。兩者都是 Apache-2.0，可由 MIT 專案引用，但正式加入時要保留各自授權與第三方聲明。

## 下一輪建議討論

1. 工作 session 是否需要 start/stop 計時器，或第一版只接受 Agent／人工完成紀錄。
2. 有效樣本的相似度與資料品質如何計分，以及 `k = 10` 是否適合作為初始先驗等效樣本數。
3. 工作日內的容量如何隨時間消耗；第一版採日級快照，之後是否需要精確工作時段。
4. `1.10`／`1.50` 壓力門檻是否經實際回測後需要調整。
5. 穩定 item ID 將來如何加入正式 report 並相容舊字串項目。
